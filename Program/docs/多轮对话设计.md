● 多轮对话实现：系统性描述                                                               
                                                                                        
  核心设计原则                                                                           
                                                                                           多轮对话的本质是：让 LLM 在每次响应时都能看到完整的对话历史，包括用户输入、模型回复、工  具调用及其结果。实现的关键在于三点：                                                   
  
  1. 消息的持久化存储                                                                    
  2. 上下文构建的完整性
  3. 工具调用结果的正确注入

  ---
  一、消息状态管理

  对话状态以 Session 对象为核心，每个 Session 维护一个 messages 数组：

  Session {
    id: string
    messages: Message[]   // ← 唯一的事实来源，贯穿整个会话
  }

  Message 的结构如下：

  type Message =
    | { role: 'user', content: string }
    | { role: 'assistant', content: string }
    | { role: 'tool', content: string, tool_call_id: string }  // 工具结果
    | { role: 'system', content: string }  // 系统提示

  二、消息的追加规则

  每条消息在合适的时机被追加到 session.messages：

  ┌───────────────────┬───────────────────────────────────────────────────┬──────────┐   
  │       时机        │                     消息类型                      │   role   │   
  ├───────────────────┼───────────────────────────────────────────────────┼──────────┤   
  │ 用户发送输入时    │ { role: 'user', content: input }                  │ 追加     │   
  ├───────────────────┼───────────────────────────────────────────────────┼──────────┤   
  │ LLM 返回纯文本时  │ { role: 'assistant', content: text }              │ 追加     │   
  ├───────────────────┼───────────────────────────────────────────────────┼──────────┤   
  │ LLM               │ { role: 'assistant', content: '' } + tool role    │ 延迟追加 │   
  │ 返回工具调用时    │ 消息                                              │          │   
  ├───────────────────┼───────────────────────────────────────────────────┼──────────┤   
  │ 工具执行完成后    │ { role: 'tool', content: result, tool_call_id }   │ 追加     │   
  └───────────────────┴───────────────────────────────────────────────────┴──────────┘   

  三、上下文构建的时机与方式

  每轮 LLM 调用前，从 session.messages 重新构建发送给 LLM 的消息列表。这个过程由
  build_llm_context() 函数完成：

  build_llm_context(session)
    → 注入 system prompt（固定在首位）
    → 按顺序取出 session.messages 中的 user/assistant 消息
    → 返回包含完整历史的 messages[]

  关键点：系统提示每轮都重新注入，而不是只注入一次。这样保证即使上下文被压缩（summarizati
  on），系统指令也不会丢失。

  四、工具调用循环

  当 LLM 返回 tool_calls 时，进入工具循环：

  while (true):
      1. 发送 ctx.messages（当前完整上下文）给 LLM
      2. LLM 返回 response（纯文本 或 带 tool_calls）
      3. if 纯文本:
             session.add_message(assistant, response.content)
             break  // 本轮结束
      4. if tool_calls:
             results = execute_tool_calls_parallel(tool_calls)
             session.add_message(assistant, response.content)  // LLM 的原始回复
             for each (tool_call, result):
                 session.add_message(tool, result.output)     // 工具结果
             continue  // 再调 LLM，LLM 看到工具执行结果后决定下一步

  五、工具消息的格式要求

  工具消息必须包含 tool_call_id，用于 LLM 将结果关联到具体的调用上下文：

  { role: 'tool', tool_call_id: 'tool_abc123', content: '{"file": "src/main.ts", "line": 
  42}' }

  tool_call_id 的值必须与 LLM 响应中 tool_calls[].id 完全一致。这是 LLM API
  的要求，不一致会导致 LLM 无法理解工具结果。

  六、完整一轮的生命周期

  用户输入: "帮我修复 bug"
      ↓
  session.add_message(user, "帮我修复 bug")
      ↓
  build_llm_context(session) → [system, user("帮我修复bug")]
      ↓
  LLM 返回 tool_calls: [{ id: 'tc1', name: 'grep', args: {...} }]
      ↓
  execute_tool_calls_parallel([tc1])
      ↓
  session.add_message(assistant, "我来搜索...")
  session.add_message(tool, "在 main.ts:42 发现问题")
      ↓
  build_llm_context(session) → [system, user, assistant, tool, ...]
      ↓
  LLM 返回 tool_calls: [{ id: 'tc2', name: 'edit', args: {...} }]
      ↓
  execute_tool_calls_parallel([tc2])
      ↓
  session.add_message(tool, "已修复")
      ↓
  LLM 返回纯文本: "已修复，改动了 x 行"
      ↓
  session.add_message(assistant, "已修复...")
      ↓
  break → 本轮结束，等待用户下一条输入

  七、错误处理与边界情况

  - 工具执行失败：tool role 消息的 content 填入错误信息，LLM
  根据错误决定是否重试或告知用户
  - LLM API 错误：本轮失败，不追加 assistant 消息，抛出异常让上层处理
  - 上下文过长：在 LLM 调用前检查 token
  数量，超过阈值触发压缩（summarization），将历史消息替换为摘要

  八、总结

  多轮对话的实现本质上是一个状态累积 + 上下文重建的循环：

  1. 状态累积：每条消息追加到 session.messages
  2. 上下文重建：每次 LLM 调用时从 session.messages 构建完整历史
  3. 工具桥接：工具结果通过 tool role 消息注入，LLM 把它当作对话的一部分
  4. 循环终止：当 LLM 不再返回 tool_calls 时，本轮对话结束，等待用户下一轮输入
